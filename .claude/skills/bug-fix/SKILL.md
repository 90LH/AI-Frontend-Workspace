# bug-fix

## 使用场景

定位并修复前端 Bug 时使用。

触发条件：
- 用户报告功能异常
- 控制台有报错
- 页面渲染异常或数据显示错误
- 性能问题

---

## 输入要求

| 参数 | 是否必须 | 说明 |
|------|----------|------|
| Bug 现象描述 | 必须 | 出现了什么问题 |
| 复现步骤 | 必须 | 如何触发该问题 |
| 报错信息 | 可选 | 控制台错误栈 |
| 预期行为 | 可选 | 正常情况下应该是什么结果 |
| 影响范围 | 可选 | 哪些页面/功能受影响 |

---

## 执行步骤

1. **理解问题**
   - 明确 Bug 现象和预期行为的差异
   - 识别 Bug 类型（逻辑错误、类型错误、异步时序、渲染问题等）

2. **定位代码**
   - 根据报错栈或功能描述，用 GitNexus 定位相关文件和函数
   - 阅读相关代码，理解当前实现

3. **根因分析**
   - 找到导致 Bug 的根本原因
   - 确认是单点错误还是涉及多个模块

4. **制定修复方案**
   - 最小化改动原则：只修改必要的代码
   - 不在修复过程中引入重构
   - 说明修复方案和理由

5. **实施修复**
   - 修改代码
   - 确认 TypeScript 无新增报错

6. **验证修复**
   - 按原复现步骤验证 Bug 已修复
   - 用 Playwright 验证相关功能无回归
   - 检查边界情况

---

## MCP 调用规则

| MCP | 使用时机 |
|-----|----------|
| **GitNexus** | 定位报错函数的引用链；分析修改影响范围 |
| **Context7** | 查询框架/库的正确用法（确认是否是 API 误用） |
| **Playwright** | 在浏览器中复现 Bug；验证修复效果；检查回归 |

---

## 输出格式

```
## 理解
[Bug 现象和根因分析]

## 计划
[修复方案说明，影响范围评估]

## 变更
[已修改的文件路径和修改内容摘要]

## 验证
[复现步骤重测结果 / 浏览器验证截图]

## 遗留风险
[同类问题是否还存在 / 未覆盖的边界情况]
```
