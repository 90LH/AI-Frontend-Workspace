# api-development

## 使用场景

新增或修改前端 API 接口调用层时使用。

触发条件：
- 对接新的后端接口
- 修改已有 API 请求参数或响应处理
- 新增 API 模块（如新业务域的接口文件）

---

## 输入要求

| 参数 | 是否必须 | 说明 |
|------|----------|------|
| 接口路径 | 必须 | 如 `/api/user/list` |
| HTTP 方法 | 必须 | GET / POST / PUT / DELETE |
| 请求参数说明 | 必须 | 字段名、类型、是否必传 |
| 响应结构说明 | 必须 | 返回数据的字段和类型 |
| 所属业务模块 | 可选 | 帮助确定文件位置 |

---

## 执行步骤

1. **分析接口信息**
   - 理解接口语义和业务用途
   - 确认请求和响应的完整类型定义

2. **检查已有 API 文件**
   - 用 GitNexus 查找同业务模块的已有 API 文件
   - 检查项目的 HTTP 请求封装方式（axios 实例、请求拦截器等）

3. **定义 TypeScript 接口**
   - 为请求参数定义 `interface XxxParams`
   - 为响应数据定义 `interface XxxResponse`
   - 放置在 `src/types/` 或接口文件同目录

4. **编写 API 函数**
   - 使用项目已有的 http 实例（不自行引入 axios）
   - 函数签名明确参数和返回类型
   - 使用 async/await

5. **在页面/组件中集成**
   - 在调用处正确处理 loading、error 状态
   - 不在 API 层做业务逻辑处理

6. **验证**
   - TypeScript 类型无报错
   - 用 Playwright 或 DevTools 确认网络请求正常

---

## MCP 调用规则

| MCP | 使用时机 |
|-----|----------|
| **GitNexus** | 查找项目 HTTP 请求封装实现；查找同模块已有 API 函数 |
| **Context7** | 查询 axios 或项目使用的 HTTP 库的文档 |
| **Playwright** | 在浏览器中触发请求，确认网络调用正常 |
