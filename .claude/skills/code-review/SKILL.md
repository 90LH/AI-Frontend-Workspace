# code-review

## 使用场景

对前端代码进行审查时使用。

触发条件：
- PR/MR 提交前的自检
- 用户请求审查某段代码
- 代码合并前的质量检查

---

## 输入要求

| 参数 | 是否必须 | 说明 |
|------|----------|------|
| 目标文件或代码段 | 必须 | 需要审查的文件路径或代码内容 |
| 变更目的 | 可选 | 这段代码要解决什么问题 |
| 关注点 | 可选 | 如性能、安全、可维护性等 |

---

## 执行步骤

1. **理解变更意图**
   - 阅读代码，理解其目的
   - 对照 AGENTS.md 的开发原则

2. **检查项目规范符合度**
   - 是否使用 Composition API + `<script setup>`
   - 是否有 `any` 类型滥用
   - 是否有重复逻辑（应提取为 composable 或 utility）
   - 是否有 Options API 残留

3. **逻辑和正确性审查**
   - 异步处理是否正确（await / catch）
   - 响应式数据是否正确使用（ref vs reactive）
   - 计算属性是否有副作用
   - 组件 Props 是否有完整类型定义

4. **安全性检查**
   - 有无 XSS 风险（v-html 使用）
   - 有无敏感信息硬编码
   - 用户输入是否有验证

5. **可维护性评估**
   - 命名是否清晰
   - 单一职责原则是否遵守
   - 组件/函数是否过于复杂

6. **给出审查结论**
   - 按严重程度分级：BLOCKER / WARNING / SUGGESTION
   - 对每条问题给出具体修改建议

---

## MCP 调用规则

| MCP | 使用时机 |
|-----|----------|
| **GitNexus** | 分析被审查代码的影响范围；查找是否有已有实现可替代 |
| **Context7** | 确认框架最佳实践（判断某种写法是否符合官方推荐） |
| **Playwright** | 验证修改后的功能在浏览器中正常运行 |

---

## 输出格式

```
## 审查结论
[整体评价：通过 / 需修改]

## 问题列表

### BLOCKER（必须修改）
- [文件:行号] 问题描述 → 建议修改方式

### WARNING（建议修改）
- [文件:行号] 问题描述 → 建议修改方式

### SUGGESTION（可选优化）
- [文件:行号] 问题描述 → 建议修改方式

## 亮点
[值得保留或推广的写法]
```
