# Component Catalog 团队规则

## 目的

定义团队级 Component Catalog 维护规则，用于约束组件分类、命名、映射、复用决策和校验流程。

本规则属于 extension 层，不保存具体项目组件清单。

## 使用场景

- 项目建立或维护 `.veaw/component-catalog/`
- 组件资产需要统一分类和命名
- 设计组件需要映射到代码组件
- 公共组件变更需要评估影响范围
- Reviewer 需要检查 Catalog 是否与源码一致

## 输入要求

| 输入 | 是否必须 | 说明 |
|------|----------|------|
| 组件条目 | 必须 | Catalog 中的组件记录 |
| 源码路径 | 必须 | 组件实际源码路径 |
| 验证来源 | 必须 | GitNexus、rg、文件读取或 Reviewer |
| 团队约束 | 可选 | UI 库、设计系统、业务域、命名规则 |

## 工作流程

1. 组件分类必须引用 `core/docs/directory.md`
2. 组件职责和接口规则必须引用 `core/docs/architecture.md`、`core/docs/coding-standard.md`
3. Catalog 条目必须包含 lastVerified 与验证来源
4. 组件新增、接口变更、迁移、删除、引用变化时触发维护工作流
5. 公共组件变更必须经过影响范围分析
6. 设计组件映射必须优先匹配已有组件

## MCP要求

| MCP | 使用时机 |
|-----|----------|
| GitNexus | 校验引用关系、影响范围、相似组件 |
| Context7 | 第三方组件库官方行为不确定时 |
| Playwright | 仅在页面实现后验证视觉和交互 |

GitNexus 不可用时，按 `core/CODEX.md` 降级策略执行。

## 输出格式

```text
## 理解
[组件条目、源码路径、团队约束]

## 分类规则
[通用组件、业务组件、页面私有组件、第三方封装组件]

## 维护规则
[新增、变更、迁移、删除、引用变化的处理方式]

## 校验规则
[GitNexus / rg / 文件读取 / Reviewer]

## 风险
[过期、重复维护、与源码不一致、待验证]
```

## 注意事项

- 不把真实组件清单写入 extension 规则
- 不复制 core 规则正文，只引用规则来源
- Catalog 是源码索引与摘要，不替代源码
- 不确定字段必须标记“待验证”
