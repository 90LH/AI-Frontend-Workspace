# Bug 修复工作流

## 触发条件

用户报告功能异常、页面报错、数据显示错误等问题。

## Agent 编排

```text
Developer
  ↓
Reviewer
```

- Developer：按 Bug 修复 Skill 定位根因、实施最小修复并验证
- Reviewer：检查影响范围、修复质量、回归风险和测试建议

## 完整流程

### 第一步：收集信息
```
必须获取：
- Bug 现象描述（发生了什么）
- 复现步骤（如何触发）

可选但有帮助：
- 控制台报错信息
- 预期行为（应该是什么结果）
- 发生频率（每次 / 偶发）
```

### 第二步：定位代码
```
工具：GitNexus
任务：
- 根据报错栈或功能描述定位相关文件和函数
- 查找函数的调用链
- 确认 Bug 是单点错误还是涉及多个模块
```

### 第三步：根因分析
```
常见 Bug 类型：
- 逻辑错误：条件判断、数据计算错误
- 类型错误：undefined/null 访问
- 异步时序：Promise 未等待、竞态条件
- 响应式问题：数据更新未触发视图更新
- 引用问题：直接修改 Props、reactive 对象解构后失去响应性

输出：确认根因（不猜测，不假设）
```

### 第四步：制定修复方案
```
原则：
- 最小化改动（不在修复中引入重构）
- 说明修复逻辑和理由
- 评估修复影响范围

输出：修复方案说明
```

### 第五步：实施修复
```
- 修改代码
- 确认 TypeScript 无新增报错
- 不修改与 Bug 无关的代码
```

### 第六步：验证
```
工具：Playwright
步骤：
1. 按原复现步骤重新操作，确认 Bug 已消失
2. 检查相关功能无回归
3. 检查边界情况

输出：验证结果截图或日志
```

### 第七步：输出总结
```
## 理解
[Bug 现象和根因分析]

## 计划
[修复方案和影响范围评估]

## 变更
[修改的文件和内容摘要]

## 验证
[复现步骤重测结果]

## 遗留风险
[同类问题是否还存在 / 未覆盖的边界情况]
```
