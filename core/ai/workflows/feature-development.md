# 功能开发工作流

## 触发条件

用户要求实现新功能时（新页面、新组件、新接口等）。

## 完整流程

### 第一步：理解需求
```
输入：用户描述的功能需求
输出：
- 功能要点（1-3条）
- 受影响的模块
- 是否有不明确的地方（有则先询问用户）
```

### 第二步：分析现有代码
```
工具：GitNexus（代码图谱分析）
任务：
- 查找同类页面/组件作为参考
- 确认可复用的 composables 和 utilities
- 确认已有 API 定义
- 确认路由注册位置

原则：理解现有结构，不猜测
```

### 第三步：查询文档（如需要）
```
工具：Context7（官方文档）
触发条件：
- 不确定框架 API 的正确用法
- 需要确认最佳实践
- 有第三方库的用法疑问

不触发条件：
- 已知的标准用法
- GitNexus 中有明确的现有实现参考
```

### 第四步：制定计划
```
输出：
- 新建文件列表（路径 + 说明）
- 修改文件列表（路径 + 修改内容）
- 高风险操作标注（如修改公共组件、路由配置）

高风险操作：执行前说明影响，等待用户确认
```

### 第五步：实现
```
原则：
- 严格按照 core/docs/coding-standard.md
- 严格按照 core/docs/directory.md 放置文件
- 最小化变更，不顺手重构
- 每个文件修改后 TypeScript 无报错
```

### 第六步：验证
```
工具：Playwright（浏览器验证）
任务：
- 访问相关页面，确认正常渲染
- 走完主要操作路径（golden path）
- 检查 Console 无报错
- 检查边界情况（空数据、加载失败）
```

### 第七步：输出总结
```
格式（来自 AGENTS.md Section 8）：
## 理解
## 计划
## 变更
## 验证
## 遗留风险
```
