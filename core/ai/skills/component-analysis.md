# component-analysis

## 目的

分析项目组件资产，输出组件资产范围、组件目录与分类、接口摘要、依赖与引用关系、相似组件、组件映射和复用决策建议。

本 Skill 是分析型 Skill，不能替代 `component-create`。

## 使用场景

- 开发前确认项目是否已有可复用组件
- 盘点组件库、业务组件或页面私有组件
- 分析公共组件修改的影响范围
- 识别相似组件和重复实现风险
- 将设计组件映射到代码组件
- 输出 Component Catalog 或组件资产报告

## 输入要求

| 输入 | 是否必须 | 说明 |
|------|----------|------|
| 分析范围 | 必须 | 全项目、组件目录、页面模块、业务域或指定组件 |
| 分析目标 | 必须 | 复用、扩展、包装、新建、拆分、影响分析或映射 |
| 组件线索 | 可选 | 组件名称、设计组件名、页面路径、功能描述 |
| 技术约束 | 可选 | UI 库、预设、团队规范、禁止修改范围 |

## 工作流程

1. 明确组件资产范围和分析目标
2. 优先使用 GitNexus 分析组件目录、引用关系、调用链、依赖和相似组件
3. GitNexus 不可用时，说明降级原因，并使用 `rg`、文件读取和 imports 分析
4. 按目录和职责分类组件：
   - 通用组件
   - 业务组件
   - 页面私有组件
   - 第三方封装组件
5. 提取组件接口摘要：
   - Props
   - Emits
   - Slots
   - Expose
6. 分析依赖关系、引用关系和影响范围
7. 识别相似组件、重复实现和可复用模式
8. 将设计组件或需求组件映射到现有代码组件
9. 输出复用、扩展、包装、新建、拆分决策建议
10. 输出 Developer 可执行任务；需要创建组件时再交给 `component-create`

## MCP要求

| MCP | 使用时机 |
|-----|----------|
| GitNexus | 组件发现、引用关系、影响范围、相似组件和依赖分析 |
| Context7 | 第三方 UI 组件库、Vue API 或组件库 API 不确定时查询官方文档 |
| Playwright | 仅在实际页面实现后验证组件渲染、交互和回归 |

只有 GitNexus 不可用或目标仓库未索引时，才允许降级到 `rg/find`、imports 分析和文件读取。

## Skill调用规则

- 本 Skill 只做分析，不创建组件
- 需要新增组件时，转交 `core/ai/skills/component-create.md`
- 需要页面实现时，转交 `core/ai/skills/vue-page-create.md`
- 需要设计稿映射时，可参考 `core/ai/skills/design-analysis.md`
- 需要审查时，转交 `core/ai/skills/code-review.md`

## Template使用规则

- 分析阶段不复制 Template
- 可用 `core/ai/templates/vue-component.vue` 判断新组件实现形态
- 可用 `core/ai/templates/vue-page.vue` 判断页面私有组件组织方式
- 可用 `core/ai/templates/composable.ts` 判断逻辑是否应抽离为 composable

## 输出格式

```text
## 组件资产范围
[分析目录、业务域、页面范围、排除项]

## 组件目录与分类
[通用组件、业务组件、页面私有组件、第三方封装组件]

## 接口摘要
[Props / Emits / Slots / Expose]

## 依赖与引用关系
[调用方、被调用方、影响范围、依赖链]

## 相似组件
[相似组件、重复点、差异点、可复用可能性]

## 组件映射建议
[设计组件或需求组件到代码组件的映射]

## 决策建议
[复用 / 扩展 / 包装 / 新建 / 拆分]

## Developer 可执行任务
[后续 Skill、文件范围、实现建议、验证建议]
```

## 注意事项

- 不修改业务代码
- 不替代 `component-create`
- 不凭空判断组件职责，必须基于 GitNexus 或已读文件
- 公共组件修改必须说明影响范围
- 页面私有组件跨模块复用前必须评估是否需要提升层级
