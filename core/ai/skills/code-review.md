# code-review

## 目的

定义前端代码审查流程，作为 Claude 与 Codex 共享的工具无关 Skill。

## 使用场景

- PR/MR 提交前自检
- 用户要求审查某个文件、代码片段或变更集
- 合并前检查规范、正确性、安全性和可维护性

## 输入要求

| 参数 | 是否必须 | 说明 |
|------|----------|------|
| 目标文件或代码段 | 必须 | 需要审查的路径或内容 |
| 变更目的 | 可选 | 这段代码要解决什么问题 |
| 关注点 | 可选 | 性能、安全、可维护性、类型等 |

## 执行流程

1. 阅读目标文件或代码段，理解变更意图
2. 如有 Git 变更，查看相关 diff
3. 检查是否符合 `core/AGENTS.md` 和 `core/docs/coding-standard.md`
4. 检查逻辑正确性、异步处理、响应式使用和类型完整性
5. 检查安全风险，包括 `v-html`、敏感信息、输入校验
6. 检查可维护性，包括命名、职责边界、重复逻辑和复杂度
7. 按严重程度输出问题，不直接修改代码，除非用户明确要求修复

## MCP 要求

| MCP | 使用时机 |
|-----|----------|
| GitNexus | 分析被审查代码的调用方、影响范围和替代实现 |
| Context7 | 确认框架或第三方库最佳实践 |
| Playwright | 用户要求验证修复后行为时使用 |

调用顺序：GitNexus -> Context7 -> Playwright。

只有 MCP 不可用或目标仓库未索引时，才允许降级到 `rg`、读取 imports、类型检查和手动验证步骤。

## 输出格式

```text
## 审查结论
[通过 / 需修改]

## 问题列表

### BLOCKER（必须修改）
- [文件:行号] 问题描述 -> 建议修改方式

### WARNING（建议修改）
- [文件:行号] 问题描述 -> 建议修改方式

### SUGGESTION（可选优化）
- [文件:行号] 问题描述 -> 建议修改方式

## 亮点
[值得保留或推广的写法]
```

## 注意事项

- 审查结论优先列问题，再给摘要
- 没有问题时明确说明未发现 BLOCKER/WARNING
- 不把风格偏好当作缺陷
- 不直接修改代码，除非用户明确要求
- 不修改 Claude 专属 `.claude/skills/*/SKILL.md`
